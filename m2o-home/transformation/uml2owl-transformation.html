Transformation of the UML model into OWL artefacts

Introduction

The Publications Office of the European Union began a project in 2015 to develop an eProcurement ontology, the motivation for which can be read in the eProcurement Ontology Architecture and Formalisation Specifications. The ultimate objective of the project is to produce a commonly agreed ontology that will conceptualise, formally encode, and make available in an open, structured and machine-readable format, data about public procurement. The data will cover procurement end-to-end, i.e., from notification, through tendering to awarding, ordering, invoicing, and finally, payment.

The process and the methodology adopted involve crafting the conceptual model in Unified Modelling Language (UML) and then, by abiding to a set of conventions and recommendations, transforming that model into a formal ontology expressed in Web Ontology Language (OWL) .

This section provides a working definition of the transformation rules for converting the UML conceptual model into the formal ontology and validation data shapes. These rules are organised in accordance with the eProcurement Ontology Architecture and Formalisation Specification.

Current Knowledge

Much has been written about transformations from UML to OWL and vice versa.

The work by Khan and Porres , transforms selected elements of UML models containing multiple UML class, object and state-chart diagrams into OWL to analyse consistency of the models. A similar approach is presented in an earlier paper by Khan et al, which focuses on detecting inconsistency in models containing UML class and state-chart diagrams.

A 2016 paper by el Hajjamy et al, and two by Zedlitz et al (2012) and (2014), investigate the differences and similarities between UML and OWL in order to present transformations of selected (and identified as useful) elements of UML class diagram. In the 2014 paper, the need for UML–OWL transformation is additionally motivated so as not to necessitate repeating the modelling independently in both languages.

Kiko’s 2005 paper compares OWL abstract syntax elements to the equivalent UML features and appropriate OCL statements. The analysis is conducted in the direction of OWL to UML. For every OWL construct, its UML interpretation is proposed.

Three further papers by Xu el al in 2009 and 2012, and Na et al in 2006, focus on extracting ontological knowledge from UML class diagrams, and describe some UML–OWL mappings with the aim of reusing the existing UML models and streamlining the building of OWL domain ontologies.

Sadowska, 2019, presents a comprehensive review of related work. It is used as a guideline for ensuring the necessary coverage of the transformation rules specified in this report. It is important to note that not all the UML elements are dealt with here, only the ones employed in the eProcurement conceptual model.

How to read this documentation

The remainder of this topic comprises four sections covering major UML topics. The section Classes and Attributes treats classes and attributes. The second section deals with the associations, dependencies and generalisations. The third section describes the transformation rules i.e., how the data types and enumerations should be transformed and finally, the fourth and final section provides transformation rules applicable to all UML elements, concerning comments, labels and notes.

Each section provides a table with an overview of the transformation rule set presented within. The table has three columns, one for every layer of the ontology architecture comprising rules: (a) , (b) , and (c) .

Transformation rules are specified in normative language and are aided by prototypical UML diagram fragments (usually preceding the rule) along with a representation of the corresponding OWL fragment depicted in graffoo visual notation. The diagrams are provided side to side to aid understanding, with UML fragment on the left constituting the source of the transformation, and the OWL fragment on the right representing the final result of the transformation.

Each transformation rule is accompanied by the formal OWL representation in Turtle and RDF/XML syntaxes corresponding to the depicted UML fragment from the preceding figure. RDF/XML is a syntax that expresses RDF graphs as an XML document. Turtle is a textual syntax for RDF that is compact and resembles a natural text form with abbreviations for common usage patterns and data types.

UML visual notation

This section provides the main UML elements employed in this document. A detailed description can be consulted in the standard specifications uml2.5 and in the UML user guide.

Figure 1. UML visual notation for classes and attributes
Figure 2. UML visual notation for data types and enumerations
Figure 3. UML visual notation for association, generalisation, and dependency

UML visual notation for classes and attributes depicts simple, abstract, and regular classes with and without attribute specifications. Note that no class methods are ever employed as this document as the transformations aim at data structures only.

UML visual notation for data types and enumerations depicts a primitive datatype and an enumeration. No complex data types are depicted as they are treated in the same manner as classes.

UML visual notation for association, generalisation, and dependency depicts association, generalisation and dependency connectors as the only ones necessary to model the eProcurement conceptual model.

Graffoo visual notation

This section provides the main Graffoo elements employed in this document. A detailed description can be consulted in the OWL standard specifications owl2 and in the graffoo user guide.

Figure 4. Graffoo visual notation for classes, instances and datatypes
Figure 5. Graffoo visual notation for object and data properties and generic links

A yellow rectangle with solid black border is used to declare classes. Solid black and labelled arrows are used to declare class axioms. A green rhomboid with solid black border is used to declare data types. Solid black and labelled arrows are used to declare class axioms.

A pink circle with solid black border is used to declare individuals. Solid black and labelled arrows are used to declare axioms and assertions among individuals.

A green solid line is used to declare data properties, where the empty circle at the beginning identifies the property domain, and the empty arrow at the end indicates the property range. A blue solid line is used to declare object properties, where the solid circle at the beginning identifies the property domain, and the solid arrow at the end indicates the property range.

The transformation rules for UML classes and attributes, connectors, data types and descriptors can be accessed via the menu items on the left.
